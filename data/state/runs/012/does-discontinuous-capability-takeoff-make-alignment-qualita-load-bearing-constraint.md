## Conjecture

**Conjecture:** Discontinuous capability takeoff makes alignment *qualitatively harder only when* safety and governance methods depend on gradual, capability-coupled feedback loops; if alignment methods are instead theory-driven and robust to large capability jumps, the discontinuity itself is not the core difficulty. What matters most for which trajectory occurs is not “speed” in the abstract but whether the underlying capability gains come from many loosely additive improvements or from a few tightly coupled bottlenecks whose resolution unlocks multiple capacities at once.

Using the load-bearing test: in many stories about continuous takeoff, the key detail is not merely that progress is slower. It is that slower progress permits iterative correction: deploy, observe failures, revise training, improve interpretability, update institutions, repeat. If you remove that gradualism, many other parts must change too. Evaluation can no longer rely on frontier deployment experience; governance can no longer count on warning shots; alignment can no longer assume that misgeneralization will surface in weaker systems before appearing in stronger ones. So “continuity” is load-bearing because it supports a whole web of compensating mechanisms.

Conversely, if one claims discontinuity is much harder, the load-bearing part is not drama about “fast change” but the loss of error-correcting structure. A sharp jump matters because alignment currently appears heavily empirical and outer-loop dependent: we learn from models by interacting with them. If capability suddenly outruns interpretability, control, and institutional response, the problem changes kind: from iterative engineering under feedback to one-shot design under deep uncertainty.

What determines which trajectory is more likely? The explanatory constraint points to system structure. Continuous takeoff is more likely if relevant capabilities scale via many semi-independent increments — data, compute, algorithms, tooling, engineering reliability, domain adaptation — so progress in one area does not automatically unlock all others. Discontinuous takeoff is more likely if there exist hidden thresholds or bottlenecks that are jointly load-bearing across many competencies: e.g., a general planning architecture, autonomous research ability, or a training regime that converts more compute into broad strategic competence. In that case, one change forces compensating changes everywhere else in the model of progress, signaling a genuine discontinuity mechanism rather than a narrative label.

Historically, “takeoff” arguments often inherit assumptions from economic growth, recursive self-improvement, or benchmark-scaling discourse without testing which details actually constrain outcomes. The useful distinction is therefore not binary continuous/discontinuous, but whether safety progress is structurally coupled to capability progress. If alignment knowledge lags and depends on deployment feedback, discontinuity is qualitatively worse. If alignment can be made more like mathematics than like trial-and-error engineering — deriving generally valid constraints in advance — then discontinuity loses much of its special threat.

So the deepest implication is: the takeoff debate is really a debate about **feedback topology**. Discontinuity is dangerous insofar as alignment currently relies on slow corrective loops, and likely insofar as capabilities are bottlenecked by a few deeply load-bearing insights.
